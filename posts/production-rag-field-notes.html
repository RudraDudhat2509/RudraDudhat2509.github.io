Production RAG, the field notes

A RAG demo takes an afternoon. A RAG system that survives real users takes everything below. These are my working notes, kept while building and debugging retrieval pipelines, published as-is because I keep re-explaining them.

Chunking: where most pipelines quietly die

Chunking is how you split documents before embedding, and it decides your ceiling: bad chunks mean bad retrieval regardless of which model you throw at it. The options, in the order I reach for them:

  • Recursive character splitting (paragraph → line → sentence → word) is the default for most systems.
  • Fixed-size (every N tokens) is fast and structure-blind. Fine for logs and CSVs.
  • Semantic chunking embeds sentences and splits where similarity drops. Best quality, expensive. Worth it for research papers and legal docs.
  • Late chunking embeds the full document first, then pools per chunk. It solves pronoun resolution, where "it" in chunk 12 refers to something defined in chunk 3.

Two rules that are non-negotiable in production: always keep 50 to 100 tokens of overlap so answers don't get split across chunk boundaries, and always store metadata (source, page, section) on every chunk. The metadata feels optional until the first time you have to debug why retrieval returned garbage.

Embeddings: treat the model like a database schema

The decision tree is shorter than the blog posts make it look. Multilingual corpus → BGE-M3 locally. Private data that can't leave your servers → local. Past roughly 4B tokens a month, local wins on cost anyway. Otherwise text-embedding-3-small is fine for fast iteration.

The part everyone learns the hard way: changing your embedding model means re-embedding the entire corpus. Treat it like a database schema migration, because that's what it is. And set normalize_embeddings=True, or your cosine similarity is quietly wrong.

Retrieval: hybrid or you will get paged

Dense vector search is semantic: it finds meaning, and it misses exact strings. Sparse search (BM25) is the opposite: it nails LangfuseError 401 and product names, and misses synonyms. Each one fails on exactly the queries the other one handles.

So production retrieval is both, merged with reciprocal rank fusion. RRF works on ranks, not scores: 1/(rank_dense+60) + 1/(rank_sparse+60). On top of that base, the upgrades that have actually earned their complexity:

  • Metadata pre-filtering: filter by version, product, or date before the vector search, never after.
  • Multi-query: three phrasings of the user's question, results unioned, run in parallel.
  • HyDE: generate a fake answer, embed that, search with it. Fake answers look more like real documents than questions do.
  • Small-to-big: index small child chunks for precision, hand the larger parent chunk to the LLM for context.
  • Contextual retrieval: prepend LLM-generated context to each chunk before embedding. Expensive at index time, but the recall gains are real.

Reranking: retrieve wide, then get picky

Your embedding model encodes documents without ever seeing the query. A cross-encoder reranker concatenates query and document and runs full attention across both, which makes it accurate and slow. The production pattern:

Retrieve top-50 fast, rerank, return top-5. Never retrieve only 5: if the correct document is at rank 6, it's gone forever.
from sentence_transformers import CrossEncoder

reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")
scores = reranker.predict([[query, doc] for doc in candidates])

Reranking costs 200 to 500ms. Cache results in Redis, since the same query produces the same ranking. And be honest with yourself: if retrieval quality is bad enough, no reranker is rescuing it.

Evaluation: RAGAS, read in causal order

Four metrics, and the order matters because you fix root causes, not symptoms:

  • Context recall: did retrieval find the needed chunks? This is the dangerous one.
  • Context precision: of what was retrieved, how much was relevant? Low means noisy retrieval: add a reranker or metadata filters.
  • Faithfulness: are the answer's claims supported by the context? High means no hallucination. It does not mean the answer is correct.
  • Answer relevancy: does the answer actually address the question?
Low recall means missing information, and the LLM must hallucinate to fill the gap. That is the primary RAG failure mode in production.

The mental model I use: precision asks "is my bag full of junk?", recall asks "did I miss anything?". Raising top_k buys recall and costs precision, which is exactly why the reranker exists. One more rule: if you generate with one model, judge with a different one. No self-scoring.

Security: the part everyone ships without

Three attacks I check for in any RAG system that takes user content:

  • Indirect prompt injection: the user's query is innocent, but a retrieved document contains instructions and the LLM follows them. Scan at index time and retrieval time, wrap retrieved content in XML tags, and escape angle brackets from untrusted content so nobody closes your tags for you.
  • Data exfiltration: access control belongs at the vector DB query level, not the application level. Unauthorized chunks must be invisible to search, not discarded after retrieval.
  • Context flooding: one oversized document fills the context window and evicts your system prompt. Hard-cap chunk size and total retrieved context.

If retrieved content can trigger tool calls, the stakes change completely: that's retrieved text causing real-world actions. Privilege separation helps (a read-only analysis pass, then a decision pass that never sees raw chunks), and anything high-stakes like payments or deletes gets a human in the loop. Trust tiers for sources: internal > verified > external > user uploads.

The whole pipeline, one line

query → multi-query (3 variants) + HyDE
      → metadata pre-filter
      → parallel dense + sparse
      → weighted RRF (top-20)
      → cross-encoder rerank
      → small-to-big expansion
      → LLM

That's the system I'd defend in a design review. Every stage exists because removing it produces a failure I've either hit or read a postmortem about.

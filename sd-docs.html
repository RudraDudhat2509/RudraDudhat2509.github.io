SD-01 Foundations

Latency Tiers

The numbers every engineer carries in their head. When an interviewer asks "is this fast enough?", these tiers are the answer. Know them cold — not just what they are, but why they're 1000x apart.

The Hierarchy

Every storage medium and network hop has a characteristic access latency. These fall into three orders of magnitude: nanoseconds, microseconds, milliseconds.

OperationLatencyRelative
L1 Cache~1 ns1×
L2 Cache~4 ns4×
RAM~100 ns100×
SSD random read~100 µs100,000×
Network — same datacenter~500 µs500,000×
HDD random read~10 ms10,000,000×
Network — cross-region~100 ms100,000,000×

The 1000× Rule (and the 100× Exception)

Each tier is approximately 1000× slower than the one above it. RAM → SSD is 1000×. SSD → HDD is 100×. Same-DC → Cross-region is ~200×.

Exception to memorize

L1 Cache → RAM is 100×, not 1000×. L1 is 1ns, RAM is 100ns. Don't say 1000× here — interviewers notice.

Why the gaps? Physical reality. L1 cache is silicon on the same die as the CPU. RAM is a separate chip on the motherboard — signals travel farther. SSD has a controller, flash cells, and a PCIe bus. HDD has a spinning platter and a mechanical arm that physically seeks. Every layer adds physical distance and mechanical overhead.

The "Is This Acceptable?" Test

Use latency tiers to reality-check design decisions fast:

  • Cache miss in Redis should be ~100µs. If it's 10ms, your Redis is overloaded or networked wrong.
  • DB query with good index: 1–10ms is normal (SSD read + network in DC). 100ms means missing index or table scan.
  • Web page load: 100ms is fast, 1000ms is noticeable, 3000ms is failure.
  • Cross-region replication lag: 100ms+ is expected. Design around it, don't fight it.

Practical Use in Interviews

An interviewer says "your search service queries Elasticsearch on every autocomplete keystroke." You say: "Elasticsearch is in the same DC, so ~500µs per query. At 5 keystrokes/second per user, that's fine. But at 10M active users simultaneously, that's 50M queries/second — we need to debounce client-side and cache top queries in Redis."

That's latency tiers being used as a reasoning tool, not a lookup table.

Gate Question

You design a feature that reads a user's settings on every API request. Settings are in PostgreSQL. At 100k requests/second, is this acceptable?

Answer: PostgreSQL read = ~1ms (SSD in DC). 100k/s is 100k concurrent queries. Postgres can handle ~5k–10k queries/sec on a single node before latency climbs. So no — cache settings in Redis (~100µs). That's a 10× latency win and removes DB load entirely.

Capacity Estimation

The opening 5 minutes of every system design interview. Not about being precise — about showing you can reason from first principles to architecture decisions. The skeleton matters more than the numbers.

The 7-Step Skeleton

Run through these in order, every time. Write numbers on the board as you go.

#StepFormula
1Writes/seconddaily_writes ÷ 105
2Reads/secondwrites/s × read:write ratio
3Size per eventestimate KB or MB per record
4Storage/daywrites/s × size × 86,400
5Total storagestorage/day × retention days
6Bandwidthreads/s × size per event
7Servers neededpeak_req/s ÷ capacity_per_server

The Most Important Anchor: 1 Day ≈ 10⁵ Seconds

86,400 seconds in a day ≈ 100,000 = 105. Not 106.

Common mistake

Dividing by 106 instead of 105. This gives numbers 10× too small. "500M tweets/day = 500 tweets/sec" is wrong. It's 5,000/sec. The anchor is 105, always.

Drill until it's muscle memory:

  • 500M events/day → 5,000/sec
  • 2B events/day → 20,000/sec
  • 50M events/day → 500/sec
  • 100M events/day → 1,000/sec

Read:Write Ratios

Product typeRatioReasoning
Social (Twitter, Instagram)100:1Every tweet is read by 100s of followers
Messaging (WhatsApp)1:1Every message sent is read once
Video streaming10:1Videos watched many times, uploaded rarely
Ride-sharing5:1Location updates written often, read for matching

Peak vs Average

Traffic is never flat. Design for peak, not average.

  • Normal products: peak = 3× average (morning/evening spikes)
  • Event-driven: peak = 10× average (ticket drops, product launches)

The Architecture Decision Table

Numbers from the skeleton feed directly into architecture choices. This is the entire point of the exercise.

SignalThresholdArchitecture implication
Req/sec< 500Single server monolith is fine
Req/sec500–10kLoad balancer + multiple app servers
Req/sec> 10kHorizontal scaling, caching, partitioning
Storage< 1 TBSingle database node
Storage1–100 TBPartitioned/sharded database
Storage> 100 TBDistributed storage (S3, HDFS, Cassandra)
Write rate> 10k/secKafka ingest + Cassandra or TimescaleDB
Bandwidth> 10 GbpsCDN mandatory

Example: URL Shortener

Given: 100M new URLs shortened per day, 10:1 read:write, 500 bytes per URL record, 5-year retention.

Writes/sec  = 100M / 10^5 = 1,000/sec
Reads/sec   = 1,000 × 10  = 10,000/sec
Size/event  = 500 bytes
Storage/day = 1,000 × 500B × 86,400 ≈ 43 GB/day
Total       = 43 GB × 365 × 5 ≈ 78 TB
Bandwidth   = 10,000 × 500B = 5 MB/sec (trivial)
Servers     = 10,000 req/s ÷ 5,000 req/s per server = 2 servers

Verdict: storage at 78 TB means we need a partitioned database. Reads at 10k/sec → need load balancer. Writes at 1k/sec → Postgres can handle it. Bandwidth is low → CDN optional.

Gate Question

WhatsApp: 1B users, 50 messages/user/day, each message 1 KB. How much storage per year?

Answer: 1B × 50 = 50B messages/day. Storage/day = 50B × 1KB = 50 TB/day. Per year = 50 × 365 ≈ 18 PB. Implication: distributed storage (Cassandra + object store), no single Postgres.

SQL & ACID Deep
SD-03 Data Layer

SQL & ACID Deep

ACID isn't a checklist — it's four different mechanisms solving four different failure modes. If you can't explain the mechanism behind each letter, you don't know ACID.

A — Atomicity

The failure mode it solves: You transfer ₹500 from Account A to Account B. That's two operations: debit A, credit B. If the server crashes after debiting A but before crediting B, ₹500 disappears.

The guarantee: all operations in a transaction succeed together, or none of them do. No partial application.

The mechanism — WAL (Write-Ahead Log): Before making any change to actual data pages, Postgres writes to an append-only log file: "I'm about to debit A by 500, credit B by 500." If the server crashes mid-transaction, on restart Postgres reads the WAL — sees the transaction was never fully committed — and reverses the partial changes. The log is the single source of truth about what happened and what needs to be rolled back.

WAL bonus

The same WAL that enables rollback is also streamed to replicas for replication. One mechanism, two uses: crash recovery + standby synchronization.

C — Consistency

The failure mode it solves: A foreign key constraint says every tweet must have a valid user_id. Without enforcement, you could INSERT a tweet pointing to user_id = 99999 which doesn't exist. Your data becomes incoherent.

The guarantee: every transaction must leave the database in a valid state — one that satisfies all defined constraints (NOT NULL, UNIQUE, CHECK, FOREIGN KEY).

The mechanism: constraint enforcement at write time. The DB checks every constraint before committing. If any constraint is violated, the transaction is rejected. Note: consistency is not automatic. You define the rules. The DB enforces them. If you don't define a constraint, the DB doesn't enforce it.

I — Isolation

The failure mode it solves: Two users buy the last ticket at the same time. Both transactions read "1 ticket remaining." Both try to buy it. Without isolation, both succeed — you've sold 1 ticket twice.

The guarantee: concurrent transactions don't interfere with each other. Each sees a consistent snapshot of the data.

The mechanism — MVCC (Multi-Version Concurrency Control): Instead of locking rows (which blocks reads), Postgres keeps multiple versions of every row, each stamped with a transaction ID. When your transaction starts, it receives a snapshot ID. Every read in your transaction sees only rows whose version was committed before your snapshot ID. Other in-progress transactions are invisible to you. No locks — everyone reads their own snapshot simultaneously.

Isolation Levels

LevelDirty readNon-repeatable readPhantom read
Read UncommittedPossiblePossiblePossible
Read Committed (PG default)NoPossiblePossible
Repeatable ReadNoNoPossible
SerializableNoNoNo
  • Dirty read: reading another transaction's uncommitted data
  • Non-repeatable read: same row gives different values on two reads in the same transaction
  • Phantom read: a re-run query returns different rows (new rows appeared mid-transaction)

D — Durability

The failure mode it solves: Transaction commits. Server crashes 50ms later. On restart, that transaction's data is gone.

The guarantee: once a transaction is committed, it's permanent. Power loss, server crash, OS failure — none of these can lose a committed write.

The mechanism — fsync: When Postgres commits a transaction, it calls fsync() — a system call that forces the OS to flush all write buffers to physical disk before returning "committed" to the client. Without fsync, data could sit in OS RAM (fast but volatile). With fsync, it's on disk before you know it succeeded. Cost: fsync is slow (~10ms). This is why database writes have measurable latency even for tiny records.

Indexing

Without an index, every query scans every row — O(n). With an index, the DB finds rows in O(log n) using a B-tree (balanced sorted tree).

Composite Index Left-Prefix Rule

An index on (sender_id, created_at) can serve queries that filter on:

  • sender_id alone ✓
  • sender_id AND created_at ✓
  • created_at alone ✗ — can't skip the first column

Why? The index is sorted by sender_id first, then by created_at within each sender_id group. Without specifying sender_id, the DB doesn't know which sender_id's created_at values you want — the index is useless for the search.

Bonus: ORDER BY

If your query is WHERE sender_id = 42 ORDER BY created_at DESC, the index on (sender_id, created_at) eliminates the sort step entirely — the rows are already in order within the index. This is a huge win at scale.

Partitioning

  • Range partitioning: rows with created_at in 2024 go to one partition, 2025 to another. Queries scoped to a date range only scan relevant partitions. Risk: if 90% of reads are for "today's data," all load hits one partition (hotspot).
  • Hash partitioning: user_id % 4 distributes rows evenly across 4 partitions. No hotspots. Trade-off: can't do range scans across partitions.

When Postgres Breaks

Postgres is excellent until it isn't. Know the ceiling:

  • Single-node ceiling: ~10–50 TB of data, ~5–10k sustained writes/sec. Beyond this, you feel it.
  • VACUUM bloat: MVCC keeps old row versions for concurrent readers. VACUUM cleans them. At high write rates (millions of rows/day updated), VACUUM can't keep up → dead rows accumulate → table bloat → everything slows.
  • Cross-shard joins: once you split data across machines (sharding), JOINs across shards require fetching data from multiple machines and joining in the application. This is painful and slow.
  • Schema migrations: ALTER TABLE ADD COLUMN NOT NULL on a 200M-row table can lock the entire table for minutes in production. You need migration strategies (add nullable first, backfill, add constraint).

Observable signals you've hit the wall: p99 query latency climbing despite good indexes. Buffer hit rate dropping below 95% (data doesn't fit in RAM). VACUUM running constantly and still losing ground. Replica lag consistently > 1 second.

NoSQL Selection
SD-03 Data Layer

NoSQL Selection

NoSQL is not "Postgres but faster." Each NoSQL database was built to solve a specific problem Postgres can't handle at scale. The choice is determined by your access pattern, not by preference.

Cassandra

Built for: massive sustained write volume (millions/sec), time-series data, multi-region deployments.

Architecture: a ring of equal nodes — no master, no single point of failure. You write to any node. That node acts as coordinator, hashing your partition key to determine which nodes actually own the data (typically 3 replicas).

Write path: the write hits a commit log first (for durability, like WAL), then an in-memory structure called a Memtable, then is flushed to disk as immutable SSTables. Cassandra never updates in-place — it appends. This is why writes are so fast: no seek, no random write, just append.

Consistency: tunable. You configure how many replicas must acknowledge a write before you call it successful. ONE = fastest, risk of stale reads. QUORUM = majority must agree, strongly consistent. ALL = slowest, all replicas must agree.

Critical rule — design tables around queries: unlike SQL where you normalize data and JOIN at query time, Cassandra requires you to model data the way you'll read it. If you need to fetch "all messages from user 42, sorted by time", your table's partition key must be user_id and clustering key must be created_at. Cassandra can't JOIN.

When to pick Cassandra

Massive write volume (IoT, activity feeds, logs). Time-series data. Known partition key (you always query by user_id or device_id). Multi-region with active-active writes. Self-hosted or want control over ops.

DynamoDB

Built for: AWS-native workloads, variable/bursty traffic, zero operational overhead.

Serverless: you don't manage servers, sharding, or scaling. Configure capacity (provisioned or on-demand), pay per request. AWS handles everything else.

Data model: tables with a primary key. Primary key = partition key (mandatory) + optional sort key. With a sort key, you can range-query within a partition: WHERE user_id = 42 AND timestamp BETWEEN t1 AND t2.

TTL: built-in item expiry. Set a TTL attribute on any item and DynamoDB automatically deletes it when expired. Perfect for sessions, caches, rate limit counters.

Streams: every write (insert/update/delete) is published to DynamoDB Streams. Lambda functions can subscribe to react to changes — build event-driven workflows without Kafka.

When to pick DynamoDB

Already on AWS. Can't afford Cassandra ops complexity. Traffic is spiky and unpredictable (on-demand mode absorbs spikes). Moderate scale (< 100k writes/sec). Need TTL or Streams natively.

MongoDB

Built for: variable/evolving schemas, hierarchical data, rich query patterns.

Document model: stores JSON-like documents. No fixed schema — two documents in the same collection can have completely different fields. Perfect when your data structure evolves rapidly (early-stage products).

Embedding vs referencing: should you store a user's address inside their user document (embedding), or in a separate collection with a reference (like a FK)?

  • Embed when data is accessed together always, doesn't grow unboundedly, and is "owned by" the parent document. Fetch in one read, no JOIN.
  • Reference when data is shared (multiple documents reference the same author), or when the sub-document can grow large (comments array).

$lookup is MongoDB's JOIN equivalent — it joins across collections at query time. Slower than embedding but necessary for shared/referenced data.

When to pick MongoDB

Schema is flexible or still evolving. Data is naturally hierarchical (product catalog with varying attributes per category). Need rich ad-hoc queries with aggregation. Don't need multi-region active-active writes.

The Decision Framework

NeedPickReject
Massive sustained writes, time-series, multi-regionCassandraPostgres (ceiling), DynamoDB (ops)
AWS-native, bursty traffic, zero opsDynamoDBCassandra (too much ops)
Flexible schema, rich queries, document modelMongoDBPostgres (schema changes pain)
Caching, ephemeral data, microsecond readsRedisAny disk-based DB
ACID, JOINs, structured relational dataPostgresAll NoSQL options
Cassandra vs DynamoDB confusion

Both look similar on the surface (distributed KV with partition keys). The differentiator is operations + scale + cloud. Very high volume + self-hosted + multi-region → Cassandra. Moderate volume + AWS + zero ops → DynamoDB.

Redis Deep

Redis is not a database replacement. It's a speed layer — RAM-resident, ~100 microsecond reads, purpose-built for the access patterns that would kill a disk-based DB under load.

Data Structures

Redis isn't just key-value. It has six data structures, each optimized for a specific pattern:

String

The basic type. Binary-safe — can hold text, JSON, or raw bytes. Key operations: SET, GET, INCR.

INCR pageviews is atomic — no race conditions. Even if 10,000 requests hit simultaneously, each gets a unique incremented value. This replaces a SELECT ... FOR UPDATE in Postgres for counters.

Hash

A map of field→value pairs, like a mini-JSON object. HSET user:42 name "Rudra" email "r@altagic.com". Read individual fields: HGET user:42 name. Update one field without rewriting the whole object. Perfect for user sessions, user profiles, configuration objects.

List

Ordered sequence with O(1) push/pop from both ends. LPUSH / RPUSH add items. LRANGE feed:42 0 49 fetches the 50 most recent items. Use for: activity feeds, recent history, message queues (basic), task queues.

Set

Unordered collection of unique strings. SADD followers:42 user_99. SISMEMBER followers:42 user_99 checks membership in O(1). SINTER finds mutual follows. Use for: tags, membership checks, deduplication.

Sorted Set (ZSet)

Like a Set but every member has a floating-point score. Members are stored sorted by score. ZADD leaderboard 9450 "rudra". ZRANK leaderboard "rudra" → rank. ZRANGE leaderboard 0 9 REV → top 10. Updates are O(log n). Use for: leaderboards, priority queues, rate limiting with timestamps as scores, trending content.

Pub/Sub

Not a data structure — a message passing mechanism. PUBLISH channel message. Subscribers on that channel receive it instantly. Fire-and-forget: if no subscriber is connected, the message is lost. For durable messaging, use a List or Kafka instead.

Eviction Policies

Redis lives in RAM. When RAM fills up, something must give. You configure what:

PolicyBehaviorWhen to use
allkeys-lruEvict least recently used key across all keysGeneral cache (default recommendation)
volatile-lruLRU, but only among keys with TTL setMixed data: some permanent, some ephemeral
allkeys-lfuEvict least frequently usedHot-key workloads (some keys 1000× hotter)
noevictionReject writes when full, return errorRedis is primary store, data loss is unacceptable

Cache Patterns

Cache-Aside (Lazy Loading)

The application manages the cache. On every read:

  1. Check Redis → hit → return data.
  2. Cache miss → fetch from Postgres → write to Redis with TTL → return data.

Pro: Only caches data that's actually read. Cache stays lean.
Con: First request always misses. If Redis is down, app still works (falls back to DB).

Write-Through

Every DB write also writes to Redis immediately. Cache is always in sync.

Pro: Cache hit rate is high from the start. No staleness.
Con: Every write is now two writes (DB + Redis). Cache fills with data that might never be read.

Write-Behind (Write-Back)

Write to Redis first, return success to user immediately. Async worker writes to DB later.

Pro: Extremely fast writes — no DB in the critical path.
Con: If Redis crashes before the async write completes, data is permanently lost. Only use if you can tolerate data loss (analytics, view counts).

TTL vs Invalidation

  • TTL: EXPIRE user:42 300 → auto-deletes in 5 minutes. Simple to implement. Staleness window = TTL value. User might see stale data for up to 5 minutes.
  • Cache invalidation: when you update the DB, also run DEL user:42 in Redis. Consistent — next read fetches fresh data. Harder to get right — you can miss invalidations, especially across services.
Rule of thumb

TTL for data where slight staleness is acceptable (feeds, product listings, leaderboards). Invalidation for data where accuracy is critical (user profile, inventory count, price).

SD-03 Data Layer

Schema Design LLD

Schema design is where interviews reveal if you can translate requirements into a data model. The process is a pipeline: entities → relationships → columns → indexes → normalization decision.

The 4-Step Process

  1. Identify entities (the nouns in the requirements). For Twitter: User, Tweet, Follow, Like, Comment.
  2. Define relationships. 1:1 (user has one profile), 1:many (user has many tweets), many:many (user follows many users, user is followed by many users).
  3. Add columns, PKs, FKs, constraints. Timestamps everywhere: created_at, updated_at.
  4. Add indexes on FK columns, search columns, sort columns.

Relationship Patterns

RelationshipImplementationExample
1:1Same table or FK in one tableUser ↔ UserSettings
1:manyFK on the "many" sideUser has many Tweets (tweet.user_id → users.id)
many:manyJunction table with composite PKUser follows User → follows(follower_id, following_id)

The Golden Rule: Never Store Derived Counts

Do not do this

tweets.likes_count INT DEFAULT 0 — This column will get out of sync. Two concurrent likes increment from the same read value. Race condition. Trust is broken.

Do this instead

Have a likes table. SELECT COUNT(*) FROM likes WHERE tweet_id = ?. This is always accurate. Cache the count in Redis if needed for performance.

Why: the raw events (like actions) are the source of truth. The count is derived from them. You always store the source, and derive the derivative. If you store both, they can diverge. The count is not data — it's a computation on data.

Twitter Schema Example

-- Core entities
users (
  id         BIGSERIAL PRIMARY KEY,
  username   VARCHAR(50) UNIQUE NOT NULL,
  email      VARCHAR(255) UNIQUE NOT NULL,
  created_at TIMESTAMPTZ DEFAULT now()
)

tweets (
  id         BIGSERIAL PRIMARY KEY,
  user_id    BIGINT REFERENCES users(id),
  content    TEXT NOT NULL,           -- max 280 chars, enforce in app
  created_at TIMESTAMPTZ DEFAULT now()
)

-- Many:many: user follows user
follows (
  follower_id  BIGINT REFERENCES users(id),
  following_id BIGINT REFERENCES users(id),
  created_at   TIMESTAMPTZ DEFAULT now(),
  PRIMARY KEY (follower_id, following_id)  -- composite PK prevents duplicates
)

-- Many:many: user likes tweet
likes (
  user_id    BIGINT REFERENCES users(id),
  tweet_id   BIGINT REFERENCES tweets(id),
  created_at TIMESTAMPTZ DEFAULT now(),
  PRIMARY KEY (user_id, tweet_id)
)

Indexes

-- Fetch a user's tweets, newest first
CREATE INDEX idx_tweets_user_created ON tweets(user_id, created_at DESC);

-- Count likes per tweet efficiently
CREATE INDEX idx_likes_tweet ON likes(tweet_id);

-- Fetch followers of a user
CREATE INDEX idx_follows_following ON follows(following_id);

Uber Schema Example

Key design decisions:

  • Driver location: a driver_locations Redis geo structure (GEOADD, GEORADIUS), not a Postgres table. B-tree can't do spatial proximity queries efficiently. Updated every 5 seconds — far too hot for Postgres.
  • Ride status: a status column on the rides table with an enum: REQUESTED → ACCEPTED → IN_PROGRESS → COMPLETED → CANCELLED. Indexed on (driver_id, status) and (rider_id, status).
  • Ratings: a separate ratings table — never a rating FLOAT column on the users table. Store the raw ratings, compute the average. Cache in Redis.
  • Fare: compute at end of ride, store in rides.fare_amount. Not computed in real-time columns.

Queues & Kafka
SD-04 Write Path

Queues & Kafka

Synchronous service calls are fragile and can't absorb traffic spikes. Queues decouple services in space and time. Kafka extends this into a durable, replayable event log.

Why Queues Exist

Without a queue, Service A calls Service B directly (synchronous). Two failure modes:

  1. Coupling: if B is down, A fails too. A user can't sign up because the email service is down.
  2. Traffic spikes: 100k signups in 1 minute hit the email service simultaneously. It falls over. All of them fail.

With a queue: A drops a message ("send welcome email to user 42"), returns success to the user immediately. Email service picks messages up at its own pace. If email service is down for 10 minutes, messages wait. No user-facing failure.

Queue vs Pub/Sub

PatternBehaviorUse case
QueueOne consumer per message. Message deleted after processing.Job processing — resize image, send email, process payment
Pub/SubEvery subscriber gets every message independently.Fan-out — new signup → analytics + email + recommendations all need to know

Kafka

What makes Kafka different from a basic queue: retention. Messages are not deleted after consumption. They stay for a configured period (default: 7 days). Any consumer can re-read them at any time.

Core Concepts

ConceptWhat it is
TopicA named category/channel for messages. user-signups, order-placed, payment-processed. Producers write to topics; consumers read from topics.
PartitionA topic is split into N ordered logs (partitions). Each partition lives on one broker. Partitions enable parallelism: 3 partitions → 3 consumers can read in parallel.
OffsetYour position in the log. Offset 500 = you've consumed 500 messages and are about to read 501. Stored per consumer group, per partition.
Consumer groupA group of consumers sharing a subscription. Each partition is owned by one consumer in the group. Multiple groups can read the same topic independently.
BrokerA Kafka server node that stores partitions and serves producers/consumers.

Why Consumer Groups Matter

Analytics service and Email service both need every user-signup event. If they share one consumer group, each message goes to only one of them. Wrong.

Solution: Analytics service has its own consumer group (offset bookmark). Email service has its own group (separate bookmark). Each group reads the full topic at its own pace. Adding a third service? New consumer group. Zero change to the producer or existing consumers.

What Happens When a Consumer Restarts

Kafka stores each consumer group's last committed offset. On restart, the consumer reads its offset from Kafka → resumes from exactly where it stopped. No messages lost (within retention period). No messages duplicated (assuming idempotent processing).

Partition count rule

Number of consumers per group ≤ number of partitions. If you have 3 partitions and add a 4th consumer, it sits idle — no partition to own. Scale partitions to scale consumers.

Event Sourcing

Traditional: store current state. accounts.balance = 1250.

Event sourcing: store every event that changed state. "Deposited 500. Withdrew 300. Deposited 1050. Withdrew 0."

Current state = replay of all events.

Why this is powerful:

  • Bug fixes via replay: found a calculation error in your fee formula? Correct the formula, replay all historical payment events → all historical balances are now correct.
  • Audit trail: every state change is recorded with timestamp and actor. Natural compliance log.
  • Time travel: what was the state of this account at 2pm yesterday? Replay events up to that timestamp.

Cost: rebuilding current state requires replaying potentially millions of events. Mitigated by periodic snapshots — save the current state every N events, so replays start from the nearest snapshot.

CDN, Replicas & Search
SD-05 Read Path

CDN, Replicas & Search

Most systems are read-heavy. The read path is a cascade of layers — each one exists because the layer below it can't scale to the required read volume alone.

CDN — Content Delivery Network

The problem: your origin server is in Mumbai. A user in New York downloads a 5MB image. Every byte travels Mumbai → Atlantic → New York. At 100ms round-trip latency, this is painful for content with many round trips.

The solution: CDN edge servers sit in cities worldwide (New York, London, Singapore, São Paulo). When New York requests an image:

  1. Request hits the New York edge server (~5ms away).
  2. Edge has the image cached → serves it immediately. 5ms total.
  3. If not cached: edge fetches from Mumbai origin, caches it, serves it. Future New York requests are fast.

Static vs Dynamic Content

TypeExamplesCDN behavior
StaticImages, CSS, JS bundles, fonts, video filesCache globally. Same file for everyone.
DynamicYour Twitter feed, account page, search resultsCannot cache globally. Must hit origin. CDN passes through.

The CDN decision rule: if two different users requesting the same URL should get the same response → static, cache it. If they'd get different responses → dynamic, origin handles it.

Read Replicas

The problem: as read traffic grows, reads hit the primary DB → primary becomes the bottleneck. But primary is also handling writes. You can't just add RAM — a single Postgres can only handle so many concurrent queries.

The solution: read replicas — exact copies of the primary database that serve read traffic. Primary handles all writes. Replicas handle reads. One primary + 3 replicas → 4× read throughput.

The mechanism: the same WAL (Write-Ahead Log) that enables crash recovery is streamed over the network to replicas. Each replica applies those WAL entries in order, staying in sync with the primary. This is called WAL streaming replication.

Replication Lag

The replica is slightly behind the primary. From the moment a write commits on primary to the moment the replica applies it: typically 10–100ms under normal load, potentially seconds if the replica falls behind.

Implications: if a user updates their profile and immediately reads it, routing that read to a replica might return stale data. Solutions:

  • Read-your-writes consistency: route reads to the primary for a brief window after the user's own write.
  • Read from primary for critical data (bank balance, inventory, anything where stale is dangerous). Read from replica for everything else.

Elasticsearch — Full-Text Search

Why Postgres LIKE '%keyword%' doesn't scale: a leading wildcard (%keyword) can't use a B-tree index. The DB scans every row. At 100M rows, this takes seconds.

The inverted index: instead of document → words (what a B-tree does), an inverted index stores word → list of document IDs that contain that word.

"garak"    → [doc_42, doc_117, doc_899]
"redis"    → [doc_3,  doc_42, doc_506]
"security" → [doc_42, doc_201, doc_899]

Query "garak redis" → intersect lists → [doc_42]

Elasticsearch is built on this data structure. Query: "find all tweets containing 'agentic security'" → look up both words in the index → intersect → return results. This is O(matches), not O(total documents).

Elasticsearch sits beside Postgres, not instead of it

Postgres is your source of truth. Elasticsearch is your search index. When a tweet is created, write to Postgres (durable) and index in Elasticsearch (searchable). Never use Elasticsearch as primary storage — it's not designed for it.

The Full Read Path Stack

User request
  → CDN edge               (static files: images, JS, CSS, video)
  → API gateway / LB       (route, auth, rate limit)
  → Redis cache             (frequently-read objects: user profiles, feeds)
  → Elasticsearch           (full-text search queries only)
  → Read replica            (most DB reads)
  → Primary DB              (writes only + critical reads)

Each layer exists because the layer below it would be overwhelmed alone. Cache misses fall through to replicas. Replica misses fall through to primary. Primary is the last resort for reads.

Gate Question

Design how YouTube serves a video to a user in Brazil. The video was uploaded from India.

Answer: On upload: transcode video → store in object storage (S3/GCS) → push to CDN edge nodes globally. On watch: user in Brazil requests video → CDN POP in São Paulo serves it from local edge cache. No traffic to India origin. Metadata (title, view count) comes from API → Redis cache → DB read replica. Comments from separate service.

SD-06 Scalability

Load Balancing & Consistent Hashing

Scale isn't just about adding servers — it's about routing traffic intelligently and distributing data so adding or removing nodes doesn't cause a cache miss avalanche.

Load Balancing Algorithms

AlgorithmBehaviorBest forWeakness
Round RobinRequests cycle 1→2→3→1→2→3Stateless servers, uniform request costIgnores server load — one heavy request can pile up
Least ConnectionsRoute to server with fewest active connectionsVariable request durations (some take 10ms, some 5s)Slightly more overhead to track connection counts
IP HashHash user IP → always same serverSticky sessions when neededHot spots if one IP range is disproportionately large

Stateless Servers Are Mandatory for Horizontal Scaling

If Server A stores your login session in its own RAM, Server B doesn't know you're logged in. Load balancer sends your next request to B → you're logged out.

The fix: servers must be stateless. Every piece of state lives in a shared external store (Redis for sessions, S3 for files, Postgres for data). Any server can handle any request because they all read from the same shared stores. Now you can add or remove servers freely.

The stateless rule

A server that dies should leave no user-visible damage. If your app breaks when you kill a random server, it has state in the wrong place.

Consistent Hashing — The Problem

You're caching data across 4 servers. Naive approach: server = hash(key) % 4. Works fine until you add a 5th server. Now hash(key) % 5 gives different results for almost every key → virtually all keys remap to different servers → massive cache miss wave → all requests hit the DB → DB falls over.

Adding or removing one server should move the minimum amount of data, not almost all of it.

The Consistent Hashing Ring

Visualize a ring of values 0 to 232. Each server is placed on the ring by hashing its identifier. Each key is placed on the ring by hashing the key. A key routes to the first server encountered when moving clockwise from the key's position.

Ring positions (simplified):
     0
  S3 ·         · S1
      \       /
       ·  K  ·
      /       \
  S2 ·         · (empty)

Key K routes clockwise → hits S1

Adding a server (S5): S5 is placed somewhere on the ring. It only takes ownership of keys between the previous server and S5's position. All other keys are unaffected. With 4 servers, adding 1 moves ~1/5 of keys (one server's worth), not all keys.

Removing a server: its keys move to the next server clockwise. ~1/4 of keys affected.

The math

With N servers, adding or removing one server remaps only 1/N of all keys. With naive modulo, a server change remaps almost all keys. The difference at scale is between a small cache warm-up and a full DB meltdown.

Virtual Nodes

Problem: with only 4 physical servers on the ring, their hash positions might be clustered unevenly. One server ends up owning 40% of the ring, another only 10%. Unequal load.

Solution: virtual nodes. Each physical server gets multiple (typically ~150) positions on the ring, each representing a virtual node. The positions are determined by hashing the server ID with different seeds: hash("server1-1"), hash("server1-2"), ..., hash("server1-150").

With 150 virtual nodes per server, the positions are statistically even around the ring. When you remove a physical server, its 150 positions are redistributed to all remaining servers — very smooth redistribution with no single server inheriting disproportionately.

SD-07 Reliability

Replication & Consensus

Systems fail. Reliability is the art of designing for failure so that when individual components die, the system as a whole keeps working — without losing data or serving contradictory state.

Sync vs Async Replication

When primary writes to disk, how does the replica get that write?

SynchronousAsynchronous
HowPrimary waits for replica to acknowledge the write before confirming to clientPrimary confirms to client immediately; sends to replica in background
Data lossZero — replica has the data before client is told "success"Up to a few seconds of writes if primary crashes before replica catches up
Write latencyHigher — adds network round-trip to replicaLower — no waiting
Default in PostgresNoYes

Choose sync for: financial transactions, anything where data loss is unacceptable. Choose async for: most applications where "replay the last few seconds of writes" is survivable.

Failover

Heartbeats: the primary and replicas send heartbeats to each other every N milliseconds (typically 100–500ms). If a heartbeat isn't received within a timeout window, the sender is suspected dead.

Failover sequence:

  1. Replicas stop receiving heartbeats from primary.
  2. After N missed heartbeats (to avoid false positives), failover is triggered.
  3. A failover controller (Patroni, etcd, or managed DB service) promotes the most up-to-date replica to primary.
  4. DNS record or load balancer config is updated to point to new primary.
  5. Other replicas re-sync from the new primary.
  6. Old primary returns? It becomes a replica of the new primary, replays WAL to catch up.

Split-Brain

The most dangerous reliability failure. A network partition splits your cluster: primary and replica can't reach each other. Both think the other is dead. Both accept writes. Now you have two primaries diverging — true split-brain. When the partition heals, you have two conflicting versions of truth with no way to automatically reconcile them.

Prevention: use consensus protocols that require a majority to elect a leader. With 5 nodes in a 3/2 partition, only the 3-node side can elect a leader. The 2-node side can't get majority → it refuses writes → no split-brain.

Raft Consensus

Raft is a consensus algorithm that ensures a cluster agrees on one leader at a time, even during network partitions and node failures.

Core concepts

  • Quorum: majority of nodes must agree on any decision. For 5 nodes, quorum = 3. For 3 nodes, quorum = 2.
  • Leader election: when no leader exists (startup or primary death), nodes vote. A node needs quorum votes to become leader. Only the node with the most up-to-date log wins — no stale node can accidentally become leader and serve old data.
  • Log replication: leader receives a write → replicates to followers → once quorum confirms receipt → commits → tells client success. If the leader dies before quorum confirms, the write is not committed and the client retries.
  • Safety: at most one leader per term (election round). Two leaders can't both get quorum simultaneously — the majority can only be in one place.
Quorum and availability trade-off

With 5 nodes and quorum of 3: tolerate 2 node failures (still have 3). If 3 fail → cluster stops accepting writes (better unavailable than inconsistent). This is a deliberate trade-off: CP not AP in CAP terms.

Circuit Breaker Pattern

The failure mode: your service calls Payment API. Payment API is degraded — each call takes 30 seconds before timing out. Without protection: every user request holds a thread for 30 seconds → threads pile up → your service runs out of threads → your service crashes → the services that call you also crash. One slow dependency cascades into a full outage.

The circuit breaker sits in front of the downstream call and tracks its health. It has three states:

StateBehaviorTransition
Closed (normal)All requests pass through. Failure rate tracked.→ Open when failure rate exceeds threshold (e.g., 50% in last 10 calls)
Open (tripped)All requests immediately return error. No calls to downstream. Downstream gets time to recover.→ Half-open after timeout window (e.g., 30 seconds)
Half-open (testing)One test request allowed through.→ Closed if success. → Open if failure.

The key insight: failing fast is better than slow failure. An immediate error to the user is better than a 30-second hang. And stopping calls gives the downstream service a chance to recover instead of being hammered while degraded.

Gate Question

You have 5 Kafka broker nodes. Two brokers die. Can the cluster still accept writes?

Answer: Yes. Quorum = 3. With 5 nodes, 2 deaths leave 3 — still a majority. Raft can elect a leader from the remaining 3. The cluster stays up. If a 3rd broker dies (only 2 left, below quorum), the cluster refuses writes to stay consistent.

SD-08 Communication

REST, gRPC & WebSockets

Services communicate through protocols. The choice of protocol determines latency, type safety, connection model, and operational complexity. Each protocol exists because the others can't do its job well.

REST

REST is not a protocol — it's a set of architectural constraints for building APIs over HTTP. The core idea: resources (nouns, not verbs). A user is a resource. A tweet is a resource. You act on resources using HTTP methods.

MethodMeaningExample
GETRead a resourceGET /users/42
POSTCreate a resourcePOST /tweets
PUTReplace a resource entirelyPUT /users/42
PATCHUpdate specific fieldsPATCH /users/42 (just the name field)
DELETEDelete a resourceDELETE /tweets/99

Status Codes

RangeMeaningKey codes
2xxSuccess200 OK, 201 Created, 204 No Content
4xxClient error400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 429 Too Many Requests
5xxServer error500 Internal Server Error, 503 Service Unavailable

Statelessness: every REST request must be self-contained. The server holds no session memory between requests. Auth token is sent in every request header (Authorization: Bearer <token>). This is what enables horizontal scaling — any server can handle any request because there's no session to maintain.

gRPC

The problem with REST JSON for internal services: JSON is text. Parsing 50 fields of JSON on every inter-service call is CPU-expensive. With 50 microservices calling each other thousands of times per second, this overhead adds up.

gRPC uses Protocol Buffers (protobuf): binary serialization format. Smaller payload (3–10× smaller than equivalent JSON), faster to serialize/deserialize. You define the API in a .proto file:

service UserService {
  rpc GetUser (UserRequest) returns (UserResponse);
}

message UserRequest {
  int64 user_id = 1;
}

message UserResponse {
  string name  = 1;
  string email = 2;
  int64  age   = 3;
}

The .proto file generates client and server code in any language automatically. Type-safe — the compiler catches mismatches. Strongly typed contracts between services.

gRPC is internal only

Browsers can't speak HTTP/2 binary frames natively. gRPC is for service-to-service communication only. Your public API must be REST. Internal: gRPC. External: REST.

WebSockets

The problem with HTTP for real-time: HTTP is request-response. Client asks, server answers. If a new message arrives for you, the server can't push it to you — it has to wait for you to ask. This forces polling: "any new messages? no. any new messages? no. any new messages? yes, here's 3." Wasteful and slow.

WebSocket: a persistent bidirectional connection. After the initial HTTP handshake (Upgrade: websocket), the connection stays open. Both sides can send data at any time without waiting for the other to ask.

Use for: chat (other user's message → server pushes to you), live sports scores, collaborative document editing (someone else moves a cursor → all editors see it), order status updates, live dashboards.

Cost: a persistent connection has a cost. Each connection holds a file descriptor. At 1M concurrent users, you need a server that can hold 1M open connections (not 1M requests/second — different scale). This is why WebSocket servers are often separate from API servers.

Webhooks

Not a persistent connection — the opposite of WebSockets. An external service POSTs to your URL when something happens. You register your URL with the service. When the event occurs, they call you.

Example: Razorpay payment succeeds → Razorpay sends POST https://yourapp.com/webhooks/payment with the payment data. You process it. No polling, no persistent connection.

Validate webhook signatures

Anyone can POST to your webhook URL. Always verify the HMAC signature that Razorpay/Stripe/GitHub sends in the request header — it proves the request actually came from them, not a spoofed request.

API Gateway

The problem: without a gateway, every microservice must independently handle auth, rate limiting, logging, SSL termination, and CORS. 50 services × 5 cross-cutting concerns = 250 redundant implementations that can drift out of sync.

API Gateway: a single entry point for all external traffic. Routes requests to the right service. Handles cross-cutting concerns once:

  • Authentication: validate JWT at gateway. Services trust the gateway's forwarded user context.
  • Rate limiting: 100 req/min per API key, enforced at the gateway before requests hit your services.
  • SSL termination: HTTPS at the gateway. Internal traffic can be HTTP (simpler, faster).
  • Logging: every request logged centrally. No per-service logging setup needed.
  • Routing: /api/users → User Service, /api/tweets → Tweet Service. Clients know one URL.

OOP, SOLID & Patterns
SD-09 LLD Deep

OOP, SOLID & Design Patterns

Low-level design is about class structure, not architecture. You're given a concrete system ("design a parking lot") and must model it in classes that are extensible, testable, and coherent.

OOP — Four Pillars

Encapsulation

Bundle data and the methods that operate on that data into one unit. Control what's accessible from outside. A BankAccount has a balance (private) and deposit() / withdraw() (public). You can't say account.balance = 1000000 directly. You must use the methods, which enforce business rules (can't withdraw more than balance). The methods are the interface; the data is hidden.

Why it matters: if balance is public, any code anywhere can corrupt it. Encapsulation means corruption can only happen through a controlled pathway you own and can audit.

Inheritance

A class can extend another class, inheriting all its attributes and methods, and adding its own. ElectricCar extends Car. Car has start(), stop(), fuel_type. ElectricCar inherits those and adds charge(), overrides start() to be silent. The "is-a" relationship — an ElectricCar IS-A Car.

When to use: genuine "is-a" relationships. When abused (inheriting for code reuse when there's no "is-a"), it creates fragile hierarchies. Prefer composition over inheritance when in doubt.

Polymorphism

Same method name, different behavior depending on the object type. Shape has area(). Circle implements it as π×r². Rectangle implements it as w×h. Code that calls shape.area() doesn't care which shape it is — the right formula runs automatically. Same method name, different behavior per type.

Why it matters: you can write code against the abstract Shape type and it works for any shape you add in the future — no if/else chains, no switching on type.

Abstraction

Hide complexity behind a clean interface. EmailSender.send(to, subject, body). You don't know if it uses SMTP, SendGrid, or AWS SES. You just call send(). The implementation is abstracted away. Abstraction is what lets you swap implementations without changing callers.

Design Patterns

Singleton

Ensures only one instance of a class exists in the entire application. Used for: logger, database connection pool, config manager — things that should be shared and consistent everywhere.

class Logger:
    _instance = None

    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance

    def log(self, msg):
        print(f"[LOG] {msg}")

# These are the same object
a = Logger()
b = Logger()
assert a is b  # True

Thread safety concern: if two threads call Logger() simultaneously before the instance is created, both might create separate instances. Fix: use a lock around the creation check, or use the module-level import pattern in Python (modules are singletons by default).

Factory

Centralizes object creation by type. Instead of new EmailNotification() scattered across 50 files, you have one place:

class NotificationFactory:
    @staticmethod
    def create(channel: str) -> Notification:
        if channel == "email":   return EmailNotification()
        if channel == "sms":     return SMSNotification()
        if channel == "push":    return PushNotification()
        raise ValueError(f"Unknown channel: {channel}")

Adding WhatsApp notifications? Add one line to the factory. No other code changes. All callers automatically get WhatsApp support. This satisfies the Open/Closed principle.

Observer

When one object changes, automatically notify many other objects. The notifying object (publisher) maintains a list of subscribers. On change, calls notify() on all of them.

class EventBus:
    def __init__(self):
        self._subscribers = {}

    def subscribe(self, event, handler):
        self._subscribers.setdefault(event, []).append(handler)

    def publish(self, event, data):
        for handler in self._subscribers.get(event, []):
            handler(data)

bus = EventBus()
bus.subscribe("user_registered", analytics.track)
bus.subscribe("user_registered", email_service.send_welcome)
bus.subscribe("user_registered", recommender.initialize)

# When user registers:
bus.publish("user_registered", {"user_id": 42, "email": "r@..."})

UserService publishes, doesn't know who's listening. Add a new subscriber → one line, no change to UserService. This is OOP's version of pub/sub.

The key you were missing: Member-borrows-Book is many:many over time. A member borrows many books. A book is borrowed by many members sequentially. This transaction needs its own class — Loan. Same pattern as Ticket in parking lot and OrderItem in e-commerce.

The transaction class pattern

Any time two entities interact in a transaction — borrow, order, park, purchase — that interaction needs its own class. Member + Book → Loan. Vehicle + Slot → Ticket. User + Product → OrderItem. Don't put the transaction's data on either entity — it belongs to the relationship itself.
